# ADR-0007: Canary deployments via Workers for Platforms Dynamic Dispatch

- Status: Accepted
- Date: 2026-05-08

## Context

ADR-0003 committed to supporting an Argo Rollouts CRD subset that includes `strategy.canary.steps[]` with `setWeight` and `pause`. ADR-0006 + the v0.1.1 implementation realized `strategy.blueGreen` (cutover) via Cloudflare's Workers Versions and Gradual Deployments API. canary was deferred because percentage-based traffic split on a single script is not the right substrate for what we actually need:

1. **Per-step state has to persist across `apply` invocations.** A canary that says `setWeight: 10 → pause 5m → setWeight: 50 → pause → setWeight: 100` cannot be expressed as a single API call. We need durable state (current step index, last advance time, current weight) and a way to advance it incrementally.
2. **Two distinct script revisions need to be addressable simultaneously.** Versions of the same script share metadata (bindings, secrets); for canary we want stable and candidate to be fully independent so a regression in v2's binding configuration cannot break v1.
3. **Future routing dimensions go beyond percentage.** Header-based routing (`X-Beta-User: 1` → canary), sticky sessions (user-id hash), and geo-routing are credible v0.1.3+ asks. Cloudflare's Gradual Deployments only does percentage random; extending to programmable routing requires a different primitive.

[Cloudflare's Workers for Platforms — Dynamic Dispatch](https://developers.cloudflare.com/cloudflare-for-platforms/workers-for-platforms/configuration/dynamic-dispatch/) addresses all three. A "dispatch namespace" holds many independent Worker scripts, and a regular Worker (the **dispatcher**) routes requests by calling `env.NAMESPACE.get(scriptName).fetch(request)` at runtime. The dispatcher is just code, so any routing logic is expressible.

## Decision

Implement `Rollout` `strategy.canary.steps[]` on top of Workers for Platforms Dynamic Dispatch, gated by an explicit `cloudflare.com/dispatch-namespace: <name>` annotation. Rollouts without that annotation continue to use v0.1.1 cutover (`blueGreen` semantics, regardless of the strategy block).

### Topology (per annotated Rollout)

```
[external request]
  → Worker route / Custom Domain
  → DispatcherWorker  (k1c-generated, lives outside the dispatch namespace)
       ├── KV: state.get(`rollout/<ns>/<name>`)  → { weight, stableScript, canaryScript, ... }
       ├── env.NAMESPACE.get('k1c--<ns>--<name>--stable').fetch(req)
       └── env.NAMESPACE.get('k1c--<ns>--<name>--canary').fetch(req)
              (chosen by weighted random)
```

### New resources

| Resource | API group | Purpose |
|---|---|---|
| `DispatchNamespace` (CRD) | `cloudflare.k1c.io/v1alpha1` | A Workers for Platforms namespace. `name` becomes the CF namespace name, prefixed `k1c-`. |

No new CRD for the dispatcher Worker or the rollout-state KV — they are k1c-managed implementation details, surfaced as derived resources at apply time. Lowering emits them; users do not write them.

### Naming conventions

| Object | Name |
|---|---|
| Dispatch namespace | `k1c-<DispatchNamespace.name>` |
| Stable script (in namespace) | `k1c--<ns>--<name>--stable` |
| Candidate script (in namespace) | `k1c--<ns>--<name>--canary` |
| Dispatcher Worker (outside namespace) | `k1c--<ns>--<name>` (replaces the script that a non-canary Deployment would have produced) |
| State KV (shared per dispatch namespace) | `k1c/rollout-state/<DispatchNamespace.name>` |
| State key (per Rollout) | `rollout/<ns>/<name>` |

### KV state schema

JSON-encoded value at `rollout/<ns>/<name>`:

```ts
{
  currentStepIndex: number;         // index into strategy.canary.steps[]
  startedAt: string;                // ISO8601, when the current canary began
  lastAdvanceAt: string;            // ISO8601, when the last setWeight took effect
  stableScript: string;             // 'k1c--<ns>--<name>--stable'
  canaryScript: string | null;      // 'k1c--<ns>--<name>--canary' or null when no canary in flight
  weight: number;                   // 0..100, percentage routed to canary
  status: 'progressing' | 'paused' | 'aborted' | 'idle';
}
```

`status: 'idle'` means there is no canary in flight; the dispatcher serves only stable. `'paused'` means the current step is `pause: {}` (no duration) and waits for explicit promote.

### Dispatcher logic (generated by k1c)

```ts
// Generated; bound at apply time with ROLLOUT_KEY, NAMESPACE binding, STATE KV binding.
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const raw = await env.STATE.get(env.ROLLOUT_KEY);
    const state = raw ? JSON.parse(raw) : null;
    if (!state || state.status === 'idle' || state.weight === 0) {
      return env.NAMESPACE.get(env.STABLE_NAME).fetch(request);
    }
    if (state.weight === 100) {
      return env.NAMESPACE.get(env.CANARY_NAME).fetch(request);
    }
    const useCanary = Math.random() * 100 < state.weight;
    const target = useCanary ? env.CANARY_NAME : env.STABLE_NAME;
    return env.NAMESPACE.get(target).fetch(request);
  },
};
```

Routing is **percentage random** in v0.1.2. Header / cookie / sticky-by-user-id routing is deferred (see "Out of scope below").

### State machine on `apply`

Pseudocode:

```
read state from KV (or initialize as 'idle')
hash desired Worker properties + entrypoint content

if state.status == 'idle':
  upload as stable, set state = { stableScript, status: 'idle', weight: 0 }
  return

if hash(desired) == hash(stable):
  // No change to the workload; just check if we're mid-canary
  if state.status in ['progressing', 'paused']:
    advance(state)            // see below
  else:
    return                    // truly nothing to do

else:
  // Workload changed -> start a new canary, replacing any in-flight one
  if state.canaryScript != null and state.status != 'idle':
    log "aborting in-flight canary"
    delete state.canaryScript
  upload new content as canary
  state = { ...state, canaryScript, currentStepIndex: 0, weight: 0,
            startedAt: now, status: 'progressing' }
  apply(state.steps[0])       // executes the first setWeight or pause
```

Where `advance(state)`:

```
step = strategy.canary.steps[state.currentStepIndex]
if step is { setWeight: w }:
  state.weight = w
  state.currentStepIndex += 1
  state.lastAdvanceAt = now
  if state.currentStepIndex >= steps.length and w == 100:
    promote()                 // see below

if step is { pause: { duration } }:
  if duration is set and (now - state.lastAdvanceAt) >= duration:
    state.currentStepIndex += 1
    advance(state)            // continue immediately
  else if duration is unset:
    state.status = 'paused'   // wait for explicit promote
  else:
    return                    // wait, will be re-evaluated on next apply
```

`promote()` finalizes a successful canary:

```
upload canary's content as stable (overwriting stable)
delete canary script
state = { stableScript, canaryScript: null, weight: 0,
          status: 'idle', currentStepIndex: 0 }
```

### CLI surface

| Command | Effect |
|---|---|
| `k1c apply -f manifest.yaml` | Runs the state machine above. Advances the canary if a duration has elapsed. |
| `k1c rollout status <ns/name>` | Prints state from KV. Shows current step, weight, last-advance-at. |
| `k1c rollout promote <ns/name>` | Forces advance past a `pause: {}` step or jumps straight to step N (with `--to-step N`). |
| `k1c rollout abort <ns/name>` | Deletes canary, resets state to `idle`, dispatcher routes 100% to stable. |

### Defaults committed in this ADR

- **Workers for Platforms is acceptable as a runtime dependency.** k1c users opting into canary explicitly via the annotation accept the metered cost (low for personal use, fixed for enterprise tiers).
- **Routing is percentage random only** in v0.1.2. Header / cookie / sticky routing is v0.1.3+.
- **The dispatcher is k1c-generated and not user-customizable** in v0.1.2. A future ADR may add `cloudflare.com/dispatcher-script: <path>` to allow user-provided dispatcher code.
- **State progression happens on `apply`.** Re-running apply (manually, via cron, via GitHub Actions schedule) is the mechanism by which canaries advance. There is no k1c daemon.

## Consequences

- **New CRD** (`DispatchNamespace`) and new derived resources (DispatcherWorker, RolloutStateKV) increase the dialect surface. Manifest portability to a real Kubernetes cluster requires installing Argo Rollouts plus stripping `cloudflare.com/dispatch-namespace` annotations.
- **At least one extra Worker is running per canary-eligible Rollout** — the dispatcher. Each user request now does at least two hops (dispatcher → target). Latency overhead is one cold start at most for the dispatcher and a cheap subrequest.
- **State is in KV; KV writes are eventually consistent.** Two near-simultaneous applies could race on `currentStepIndex`. Mitigation: KV write keys are deterministic; in practice, the only writer is `apply`, and users do not run apply concurrently. We do not need a lock for v0.1.2; we add one if real usage shows races.
- **Promote / abort require operating on KV state separately from the manifest.** `k1c rollout` subcommands talk to KV directly. The manifest does not encode "the canary is currently at step 2" — that is operational state, not desired state.
- **Workers for Platforms required.** Users who do not want a Workers for Platforms namespace cannot use canary; they get blueGreen cutover or, if they use `strategy.canary.steps`, a warning that falls back to cutover. (This is the v0.1.1 behavior we already ship.)
- **Two-script-per-rollout cost.** During an active canary, both stable and canary scripts exist. CF account script limits apply.
- **Dispatcher upgrades.** When k1c changes the dispatcher template (bug fix, new routing logic), `apply` re-uploads the dispatcher. Diff is on dispatcher source content.

## Out of scope (will get follow-up ADRs)

- **Header / cookie / sticky-session routing.** Requires the dispatcher to consume `request.headers` and a routing rule schema. ADR will define how to express routing rules in `Rollout.spec.strategy.canary` (probably `setWeight` plus optional `match: { headers: { 'x-beta': 'true' } }`).
- **User-provided dispatcher code.** Annotation `cloudflare.com/dispatcher-script: <path>` opts out of generation; user takes responsibility for the dispatcher contract.
- **Locking on KV state writes.** If concurrent apply becomes a real workflow, add Durable Objects-backed locking.
- **Rollback to a specific prior version.** `k1c rollout undo` requires keeping a history. Out of scope; only abort-current-canary is provided in v0.1.2.

## Alternatives considered

- **Stay on Gradual Deployments for canary.** Rejected: `pause: { duration }` and step-by-step advancement aren't expressible; cannot evolve to header/cookie routing later. Forcing canary into a single API call would require k1c to spin and sleep during apply, which is undesirable for a CLI.
- **Build a Cloudflare Workflows-driven canary controller.** (See `docs/future-considerations.md`.) Workflows would let canary advance autonomously with `step.sleep(duration)`. Rejected for v0.1.2: still requires dispatcher logic, and adds the runtime mode question (apply via CLI vs Workflow). Both axes can be combined later — Workflows can host the same state machine.
- **User writes their own dispatcher.** Rejected as default: kills turnkey UX. Re-introduced as escape hatch in v0.1.4.
- **Run k1c as a control plane on Cloudflare itself.** Out of scope per ADR-0002. Workflows is the closest version of this and is in `docs/future-considerations.md` for later.

## Implementation phases

1. **v0.1.2-α** — `DispatchNamespace` CRD + provider (create / list / get / delete; namespaces are nearly immutable).
2. **v0.1.2-β** — Worker provider extended to upload to a dispatch namespace when annotation is present (`scripts.update` with `dispatch_namespace`).
3. **v0.1.2-γ** — Dispatcher template generated, deployed, and bound to the rollout-state KV. Dispatcher source is hashed for diff.
4. **v0.1.2-δ** — `Rollout` lower path: emits stable + canary worker DesiredResources + dispatcher + KV state seed.
5. **v0.1.2-ε** — State machine on apply (read KV, decide step, write KV, execute API calls).
6. **v0.1.2-ζ** — `k1c rollout {status,promote,abort}` CLI commands.

Each phase is shippable independently; α-γ get the substrate without canary semantics, δ-ζ wire it up.
