# ADR-0004: Stateless reconciler against the Cloudflare API; no cdktf/pulumi

- Status: Accepted
- Date: 2026-05-08

## Context

The translator (ADR-0002) needs to compute a diff between the desired state (manifest) and the actual state (Cloudflare) and apply it. Three implementation paths were considered:

1. **CDKTF** — generate Terraform JSON, hand off to `terraform apply` with the Cloudflare provider.
2. **Pulumi** — programmatically construct resources via the Pulumi engine.
3. **Direct SDK** — call the official `cloudflare` TypeScript SDK directly with a custom reconciler.

CDKTF and Pulumi both require:

- A separate state file (`tfstate` or Pulumi's checkpoint), making `tfstate` a second source of truth alongside the manifest.
- A heavy runtime dependency (the `terraform` binary plus provider plugins, or the Pulumi engine ~100MB).
- Five hops on every apply: manifest → k1c → CDKTF graph → tfstate → terraform → Cloudflare provider → Cloudflare API.

The state-file model also conflicts with how Kubernetes itself works. In Kubernetes, the API server is the source of truth and the client is stateless (`kubectl apply` reads desired, queries actual, sends a diff). Bolting `tfstate` onto a k8s-style UX produces two competing state stores, with predictable drift between them.

## Decision

Reconciler architecture:

- **No state file.** Cloudflare itself is the source of truth.
- **Direct dependency on the official `cloudflare` TypeScript SDK** (`@cloudflare/cloudflare`). No CDKTF, no Pulumi, no Terraform binary.
- **Ownership is encoded in the resource itself**, not in a side-channel state store. Each managed resource carries a `managed-by=k1c` marker plus a logical identity (`k1c.io/namespace`, `k1c.io/name`) using whatever side-channel each Cloudflare resource type provides:
  - Worker scripts: name prefix and Worker `metadata` (script tags) when available.
  - DNS records: `comment` field with structured key-value content.
  - R2 buckets: name prefix.
  - KV namespaces: title prefix.
  - Durable Object classes: scoped to their owning Worker, no extra marker needed.
  - Routes: zone-scoped listing plus pattern matching.
- **Apply lifecycle:**
  1. Parse and validate the manifest (zod schema).
  2. Query Cloudflare for all resources matching `managed-by=k1c` within the manifest's namespace scope.
  3. Compute create/update/delete/no-op operations.
  4. Topologically sort by declared and implicit dependencies.
  5. Execute via the SDK with retry-on-recoverable-error.
- **Per-resource state during update** is reconstructed by calling `Read` immediately before `Update`, not stored locally. This matches the formae model (ADR-0006) and makes drift visible in the diff itself.

## Consequences

- **Single dependency** for the reconciler core: the `cloudflare` npm package.
- **No two-state-file class of bug.** Drift between manifest and Cloudflare is the only diff that exists.
- **No transactional rollback.** If `apply` fails midway, the cluster is in a partial state. Mitigations: per-step idempotency, recoverable-error retry, explicit `--rollback` mode that re-applies the last known-good manifest from git.
- **`delete` requires garbage collection by ownership query.** Removing a resource from the manifest is detected by "exists in cluster with our label, missing from manifest" rather than "exists in tfstate, missing from manifest".
- **Provider schema updates require a manual SDK bump.** The official SDK is generated by Stainless and updates regularly; this is acceptable.
- **Heavy IaC features we lose:** transactional plan/apply, automatic dependency graphing, free drift detection. The first we accept; the second and third we implement ourselves at limited scope.

## Alternatives considered

- **CDKTF + Cloudflare provider.** Rejected — see Context.
- **Pulumi.** Rejected — see Context.
- **Alchemy as the runtime.** Considered. Alchemy already has the stateless idempotent-apply shape we want. Rejected for v0 because depending on Alchemy means inheriting its release cadence, its provider coverage decisions, and its TypeScript-DSL bias. Revisit if direct-SDK reconciler becomes a maintenance burden.
- **Wrangler as the deployment driver.** Rejected — wrangler is Worker-centric and does not cleanly handle R2, DNS, KV, or multi-Worker apps as a unit.
